# ROVE Hire

Internal recruitment web app for posting jobs, tracking candidates, scheduling interviews, and generating offer/NDA PDFs.

## What it does

- Sign in with email + password (JWT + bcrypt).
- Dashboard of candidates with search and status filtering.
- Create and manage job openings (title, markdown description, required skills, open/closed status).
- Closed job openings remain visible but cannot accept new candidates.
- Add candidates manually: upload a PDF resume, pick a job, and copy a secure magic link.
- Public magic-link form for candidates (phone, location, current role, notice period, salary expectation, LinkedIn). Links expire after 14 days and are single-use.
- Schedule interviews (date/time, type, interviewer, notes); status moves to Interview Scheduled.
- Mark interviews completed with hire/no-hire/maybe feedback.
- Generate offer letters and NDAs as downloadable PDFs; status moves to Offer Sent.
- Hire or reject candidates with required reasons; all actions appear on the candidate timeline.
- Resumes and generated PDFs persist across server restarts.

## Tech stack

| Layer | Choice | Why |
|-------|--------|-----|
| Frontend | React + Vite, React Router, Tailwind CSS, React Hook Form, TanStack Query, Axios, Context API | Fast dev builds, familiar React patterns, and explicit client-side state management. |
| Backend | Express.js on Node.js | Lightweight, easy to mount static frontend and API on the same origin in production. |
| Database | MongoDB with Mongoose | Flexible schema for candidates, jobs, interviews, and documents. |
| Auth | JWT access tokens + bcrypt password hashing | Simple stateless auth for a single HR user role. |
| PDFs | Puppeteer (HTML → PDF) with pdf-lib fallback for light merging | Produces realistic document headers, bodies, and signature blocks. |
| File storage | Local `uploads/` directory served statically | Keeps the app self-contained without cloud object storage. |

**Note on Next.js:** The original assignment brief suggested Next.js. We intentionally chose React + Vite + Express to keep the frontend and backend as separate, clearly bounded layers and to avoid Next.js-specific hosting/runtime requirements while still delivering a modern SPA.

## Running locally

1. Copy and fill in environment files:

```bash
cp .env.example .env
cp client/.env.example client/.env   # already points VITE_API_URL to /api
cp server/.env.example server/.env   # if present
```

Required variables:

```env
NODE_ENV=development
PORT=5000
MONGODB_URI=mongodb://localhost:27017/rove_hire
JWT_SECRET=change-this-to-something-long-and-random
UPLOAD_DIR=uploads
CLIENT_URL=http://localhost:5173
```

2. Install dependencies and seed the database:

```bash
npm run build   # installs client + server deps and builds the React app
npm run seed    # creates the HR user and sample jobs/candidates
```

3. Start the backend:

```bash
npm start
```

4. In development, start the Vite dev server separately:

```bash
cd client
npm run dev
```

Then open the client URL (default http://localhost:5173) and sign in with the credentials above.

## PDF generation

Server-side PDFs are generated by Puppeteer rendering Handlebars-like HTML templates, then printing to PDF. This gives real document styling (header, body, signature block, page breaks). If Puppeteer/Chromium is unavailable, `pdf-lib` is used as a fallback for simple document assembly.

## Hosting / deployment

- **Render:** A `render.yaml` is included. Create a Render account, connect the repo, and set `MONGODB_URI` and `JWT_SECRET` in the service dashboard. The build command is `npm run build` and the start command is `npm start`.
- **Docker:** A `Dockerfile` is included that installs Chromium dependencies for Puppeteer, builds the client, and starts the server on port 5000.
- In production the Express server serves the built React app from `client/dist` and falls back to `index.html` for React Router routes.

- E-signature integration for signed offer letters/NDAs.

- Observability: structured logging, metrics, and alerting.

## Production concerns

- JWTs are currently stored in `localStorage`, which is vulnerable to XSS. A production app should use httpOnly cookies with CSRF protection and short-lived access tokens.
- Resumes and generated PDFs are stored on disk, so the hosting environment must provide persistent disk (Render Disk, Docker volume, etc.) or switch to object storage.
- PDF generation relies on Puppeteer + Chromium; verify Chromium availability in the target runtime or switch to a pure-JS PDF library if Chromium is restricted.
